# CLAUDE.md

Guidance for Claude Code (or any AI coding agent) working in this repository.

## What this is

A monorepo for PulseQueue — positioned as **notification infrastructure**, not a notification CRUD app: any producer with a valid API key hands off a `DomainEvent` (e.g. `expense.created`, `donation.created`) onto a RabbitMQ topic exchange and PulseQueue owns everything from there — queueing, retry with backoff, dead-lettering, Redis-backed dedup, per-source rate limiting, and fan-out to Email/Push (simulated) and WebSocket (real) — while every outcome (processed, failed, retried, dead-lettered, duplicate) is persisted to Postgres and surfaced on a live, Kibana-style dark dashboard, polling REST stats and subscribing to a STOMP topic for real-time events. Renamed from EventFlow to PulseQueue to make that infrastructure framing explicit — see the root README's architecture section for the pitch.

```
pulsequeue/
├── backend/    Spring Boot 3 API (Java 21, RabbitMQ, Redis, PostgreSQL/Flyway, WebSocket/STOMP)
├── frontend/   React + TypeScript dashboard (Vite, Tailwind, TanStack Query, STOMP.js)
├── monitoring/ Prometheus scrape config + Grafana provisioning (datasource + one dashboard)
├── docker-compose.yml   Orchestrates db + rabbitmq + redis + api + web + prometheus + grafana
└── .github/workflows/ci.yml   backend (Maven+Testcontainers) + frontend (npm) jobs
```

**This repo is deliberately standalone** — it exposes `POST /api/v1/events` and `/api/v1/events/simulate` to publish sample events itself, rather than being called by CashPilot/Social Supply/PulseHub. Don't add cross-repo integration code (e.g. an HTTP client back to those repos) unless explicitly asked; the point of this repo is to demonstrate the event-driven pattern in isolation.

## Running things

```bash
cd backend && mvn -B clean test        # unit tests + Testcontainers integration test (needs Docker)
cd frontend && npm ci && npm run lint && npm run build   # or: npm run dev
```

Full stack via Docker (from repo root): `docker compose up --build` — dashboard at `:3000`, API at `:8080`, Swagger at `:8080/swagger-ui.html`, RabbitMQ management UI at `:15672` (pulsequeue/pulsequeue), Prometheus at `:9090`, Grafana at `:3001` (admin/admin, anonymous viewer access also enabled).

**Environment note:** this repo was scaffolded in a sandbox with no local JDK/Maven and a Docker daemon that wouldn't start non-interactively (see the sibling [java-patterns-lab](../java-patterns-lab) repo's memory notes for the same issue). The backend was verified via GitHub Actions CI rather than locally — `ubuntu-latest` runners have a working Docker daemon out of the box, which the Testcontainers-based integration test needs. The frontend *was* verified locally (`npm run build`, `npm run lint`, and a dev-server render check) since a local Node install was available. If a future session has working local Docker/Maven, prefer running things for real over trusting this note forever.

## The event pipeline

`DomainEvent` (`event/DomainEvent.java`) is a record: `eventId` (UUID, generated by `DomainEvent.of(...)`), `eventType`, `sourceService`, `payload` (a `Map<String, Object>`), `occurredAt`. `EventPublisher` publishes it to `pulsequeue.exchange` with the event's own `eventType` as the routing key; `pulsequeue.events.queue` is bound with `#` (catches everything) since there's a single processing pipeline, not per-type routing.

**Retry + Dead Letter Queue** (`config/RabbitMqConfig.java`): the main queue is declared with `x-dead-letter-exchange` pointing at `pulsequeue.exchange.dlx`. The listener container factory wraps `EventConsumer.onMessage` in a stateless `RetryOperationsInterceptor` (exponential backoff, `pulsequeue.retry.*` properties) that retries the *same delivery* in-process; once attempts are exhausted, `RejectAndDontRequeueRecoverer` rejects the message and RabbitMQ's dead-letter routing takes over automatically. `DeadLetterConsumer` listens on the DLQ purely to flip the corresponding `ProcessedEvent` row to `DEAD_LETTERED` — it's the *only* place that status is ever set.

**Deduplication is check-before / mark-after-success** (`service/DeduplicationService.java`), not claim-then-process. If the Redis key were claimed before dispatch, every in-process retry of a currently-failing delivery would see its own claim and get skipped as a "duplicate" instead of actually retrying. The key is only written once `NotificationDispatchService.dispatch(event)` succeeds — read the class javadoc before changing this, it's a real bug that was caught and fixed during initial development, not a stylistic choice.

**`ProcessedEvent`** (`entity/ProcessedEvent.java`) is upserted (found-or-created by `eventId`) on every delivery attempt, not just the final one — `retryCount` increments on each `markFailed` call, so the dashboard can show live retry counts while an event is still being retried, not just after it settles.

**Persistence for a failed attempt goes through `ProcessedEventRecorder`, a separate `@Transactional`-per-method bean — not a `@Transactional` `process()` method.** `EventProcessingService.process()` deliberately rethrows the triggering exception after recording a failure, so Spring AMQP's retry interceptor sees it and retries/eventually dead-letters the message. If `process()` itself were `@Transactional`, Spring would roll back *everything* written during that call — including the "this attempt failed" row — the instant the exception left the method, because that's what `@Transactional` rollback-on-uncaught-`RuntimeException` does. This was a real bug caught via CI: retries kept resetting `retryCount` to 1 instead of incrementing, and the row never actually existed by the time the DLQ consumer went looking for it (the "Event dead-lettered..." log line fired regardless, since it runs before the DB call — a log message is not proof a write happened). Calling through `ProcessedEventRecorder` (a distinct Spring-proxied bean, so each `find/recordProcessed/recordFailed/recordDuplicate` call is its own independently-committed transaction) fixed it. Don't move that logic back into one `@Transactional` method without re-solving this.

**Simulating a failure deterministically:** `NotificationDispatchService.dispatch` throws immediately if `event.payload().get("simulateFailure") == true`, guaranteeing the event exhausts retries and lands on the DLQ. This is what the dashboard's "Simulate failure" button and `PulseQueueIntegrationTest`'s DLQ test both use — don't rely on random/flaky failures to exercise this path.

## Security

`ApiKeyAuthFilter` (`security/`) guards only `POST /api/v1/events*` — everything else (dashboard reads, actuator, Swagger) is `permitAll()`. It's a plain `OncePerRequestFilter` checking a shared-secret `X-API-Key` header against `pulsequeue.security.api-key` (`API_KEY` env var), not a `UserDetailsService`/login flow — this is "only trusted producers can publish," not user auth, and there's deliberately no session, no JWT, no login screen. `SecurityConfig`'s `SecurityFilterChain` itself just `permitAll()`s everything and wires the filter in before `UsernamePasswordAuthenticationFilter`; the actual 401-and-short-circuit enforcement lives inside the filter, not in `authorizeHttpRequests(...)` rules. If you add a new write endpoint that should be producer-only, extend `ApiKeyAuthFilter.isProtected(...)`, don't bolt on a second auth mechanism. The frontend dashboard sends the same (documented-as-demo-only) dev key from `lib/api.ts` — it is the "trusted internal caller" for the Simulate buttons, not a model for how a real credential should reach a browser.

## Backend conventions (`backend/`)

- Package layout: `config` (RabbitMQ/WebSocket/typed `@ConfigurationProperties` records) → `event` (wire model) → `producer`/`consumer` → `notification` (channel fan-out) → `service` (dedup, rate limit, dashboard stats, orchestration) → `entity`/`repository` (Postgres) → `controller` (REST) → `metrics` (Micrometer).
- Config is typed, not `@Value`-injected: `RabbitMqProperties`, `RetryProperties`, `DedupProperties`, `RateLimitProperties`, `CacheProperties` are all `@ConfigurationProperties` records, enabled via `@ConfigurationPropertiesScan` on the main application class.
- Schema is owned by **Flyway** (`backend/src/main/resources/db/migration/V*.sql`), `ddl-auto: validate` — Hibernate never mutates the schema.
- Tests: JUnit 5 + Mockito unit tests per service with real branching logic (`EventProcessingServiceTest`, `NotificationDispatchServiceTest`, `RateLimitServiceTest`, `DeduplicationServiceTest`, `DashboardStatsServiceTest`), a `@WebMvcTest`+`spring-security-test` slice test for the API-key filter (`EventControllerSecurityTest`), plus one Testcontainers `PulseQueueIntegrationTest` proving the full pipeline against real Postgres/RabbitMQ/Redis (normal processing, retry-to-DLQ, and duplicate-delivery paths — this one bypasses HTTP/security entirely by calling `EventPublisher` directly).
- API-key auth (see "Security" above) guards the publish endpoints only — there's still no user login/JWT, and that's intentional; don't add one unless asked.

## Frontend conventions (`frontend/`)

- Vite + React 18 + TypeScript, path alias `@/*` → `frontend/src/*`. No React Router — it's a single dashboard page, not a multi-page app.
- `src/hooks/useDashboardStats.ts` and `useRecentEvents.ts` poll REST via TanStack Query (`refetchInterval: 3000`); `useLiveNotifications.ts` is a plain `useEffect` + `@stomp/stompjs`/`sockjs-client` subscription to `/topic/notifications`, kept outside the query cache since it's a push feed, not a fetch-and-cache resource — same split PulseHub uses between TanStack Query and its socket state.
- `vite.config.ts` needs `define: { global: "window" }` — `sockjs-client` references Node's `global`, which doesn't exist in a browser; losing this breaks the app at import time.
- `src/vite-env.d.ts` (`/// <reference types="vite/client" />`) must exist for `import.meta.env.VITE_*` to type-check — easy to lose if the frontend is ever regenerated from scratch rather than edited in place.
- **Dark, Kibana-style theme is intentional, not a placeholder** — `tailwind.config.js`'s `canvas`/`panel`/`panelborder`/`muted`/`accent`/`status.*` tokens and `index.css`'s forced `color-scheme: dark` are the whole visual identity ("ops tool," not "admin CRUD panel"). Don't reintroduce light-mode/adaptive theming or swap back to generic `slate-*`/`white` Tailwind classes without discussing it — that was a deliberate pivot away from the original light dashboard.
- `StatsGrid.tsx` renders the six primary tiles in a fixed order (Messages Published, Messages Processed, Dead Letter Queue, Retry Queue, Consumers, RabbitMQ Status) plus a secondary row (Queue Depth, Failed Attempts, Duplicates Skipped) — that specific set and order is a deliberate product decision, not just "whatever fields the API returns," so don't reorder or drop tiles casually when the API shape changes.
- `lib/api.ts` attaches a demo `X-API-Key` header (`VITE_API_KEY`, default matching the backend's dev key) to every request via the shared axios instance — the backend only actually checks it on `POST /api/v1/events*`, so it's harmless on the GET-only dashboard/stats calls.

## Things to watch for

- If you add a new event type or notification channel, it flows through the *existing* `NotificationChannel` list (`List<NotificationChannel>` constructor-injected into `NotificationDispatchService` — Spring collects all beans of that type automatically). Add a new `@Component implements NotificationChannel`, don't touch the dispatch service itself.
- `AmqpAdmin.getQueueInfo(queueName)` (used by `DashboardStatsService` for queue depth **and** consumer count, via `QueueInformation.getMessageCount()`/`getConsumerCount()`) does a passive queue declare over AMQP — it does **not** need the RabbitMQ management HTTP API/plugin, unlike a naive implementation that calls the `:15672` management API directly. Don't "fix" this by adding an HTTP client to the management API; it would be strictly worse (extra network hop, extra auth, extra failure mode) for the same data.
- `rabbitMqStatus` in `DashboardStatsResponse` is derived from the *same* `AmqpAdmin.getQueueInfo(...)` call already used for queue depth/consumer count — if it throws, status is `"DOWN"`, else `"UP"` — rather than injecting Actuator's `RabbitHealthIndicator` bean directly. That was tried first and reverted: `RabbitHealthIndicator` isn't reliably available as a directly-autowirable bean in this Boot 3.3 auto-config chain (`NoSuchBeanDefinitionException` at context startup, caught by `PulseQueueIntegrationTest`), and deriving status from the admin call already in use is simpler and arguably more meaningful anyway (it's the exact call the rest of this service depends on working). Don't reintroduce the `RabbitHealthIndicator` dependency without re-solving why it wasn't found.
- The Grafana dashboard's panels only query metrics this service actually exposes (`pulsequeue_events_*` from `EventMetrics`, plus Micrometer's free `http_server_requests_seconds_count`). There is deliberately no RabbitMQ-specific Prometheus panel — that would need the `rabbitmq_prometheus` plugin enabled and a second scrape target, which isn't wired up. Queue depth is shown on the React dashboard instead (via `AmqpAdmin`), not Grafana.
